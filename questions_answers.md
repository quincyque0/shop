# Ответы на вопросы об архитектуре микросервисов

## 1. Какие критерии использовались при разделении на микросервисы?

Разделение выполнено по принципу **bounded context** (DDD): каждый сервис отвечает за свою предметную область — каталог, заказы, уведомления.  

- Независимость развёртывания
- Единая ответственность (Single Responsibility)
- Минимизация связей между сервисами

## 2. Почему выбран именно этот стек технологий?

| Компонент | Выбор | Причина |
|-----------|-------|---------|
| Язык | Python + FastAPI | Высокая скорость разработки, встроенная поддержка async/await |
| СУБД | PostgreSQL | Надёжность, поддержка транзакций |
| Межсервисное взаимодействие | gRPC | Эффективный бинарный протокол |
| Брокер сообщений | RabbitMQ | Зрелое, проверенное решение |

## 3. Как сервисы справляются с ошибками?

- **Order Service** — перехватывает `grpc.RpcError` и возвращает HTTP 503 при недоступности Catalog Service
- **RabbitMQ** — подключение через `connect_robust` с автоматическим переподключением
- **Notification Service** — использует цикл с `await asyncio.sleep(2)` для ожидания доступности брокера

## 4. Как организован деплой?

**Локально:** `docker compose up --build` (или `make up`)

**CI/CD:** GitHub Actions с 6-стадийным пайплайном:

| Стадия | Инструмент | Описание |
|--------|-----------|----------|
| Lint | flake8, hadolint, helm lint | Проверка кода, Dockerfile, Helm |
| Build & Test | docker compose | Сборка + интеграционные тесты |
| Security Scan | Trivy | Сканирование образов на уязвимости |
| Publish | docker/build-push-action | Push в ghcr.io |
| Deploy Staging | Helm | Деплой в staging namespace |
| Deploy Production | Helm | Деплой в production (manual approval) |

**Kubernetes:** Helm-чарт с отдельными values для staging и production

## 5. Что было самым сложным?

1. Обеспечение корректного порядка запуска сервисов (healthcheck и `depends_on` в docker-compose)
2. Интеграция gRPC-сервера в lifespan контекст FastAPI-приложения

## 6. Что можно улучшить?

- Распределённый трейсинг (OpenTelemetry)
- Централизованный сбор логов (ELK-стек)
- Circuit breaker для gRPC-вызовов
- Полноценная отправка email (не заглушка)
